The Problem

Most teams using Claude Code treat every session like a blank slate. They paste in context, re-explain coding conventions, correct the same mistakes, and hope for consistency. It is like onboarding a new contractor every morning who forgets everything by evening.

What "default" looks like

You open Claude Code, describe what you want, and Claude writes code using its training. The code works, but it does not follow your naming conventions, it uses patterns you have already rejected, and it forgets the decision you made last Tuesday about the data model.

What "production" looks like

You open Claude Code, and it already knows your stack, your conventions, your project history, and why you chose polling over WebSockets. It asks for confirmation before invoking tools, and it remembers that you hate em-dashes. You spend your time on the problem, not on context-setting.

The core insight: Claude Code is not just a chat interface. It is a configurable system with a file-based architecture that lets you persist context, enforce rules, and create reusable capabilities. Most users never touch these features.

Layer 1

CLAUDE.md: The Governance File

This is the most important file in your setup. It loads automatically at the start of every conversation and tells Claude who you are, how your project is structured, and what rules to follow. Think of it as an always-on briefing document.

What belongs here

  • Directory mapA quick reference so Claude knows where things live without exploring.
  • Governance table"For this task, load this one file." Prevents Claude from loading everything at once and burning context.
  • Hard rulesCoding standards, communication preferences, architectural principles that apply everywhere.
  • Skills indexA table of available skills so Claude can suggest (not auto-invoke) the right one.

What does NOT belong here

API documentation, code examples, detailed architecture descriptions, or anything longer than a few sentences per topic. CLAUDE.md loads every session. Every line costs context tokens, and context tokens are your budget for actual work.

Rule of thumb: Keep it under 200 lines. If it is getting longer, the content probably belongs in a skill, a rule file, or a workflow document that loads only when needed.

Hierarchy

CLAUDE.md files cascade like CSS. A file at the project root sets global rules. A file in a subdirectory can add or override rules for that context. A file in _codeBase/ignition/ adds Ignition-specific rules without repeating the global ones.

5
Layer 2

Path-Scoped Rules: Convention Enforcement

Rules in .claude/rules/ activate automatically when Claude edits files matching a path pattern. They are invisible to the user and require no invocation. This is where coding conventions go, not in CLAUDE.md.

// .claude/rules/jython.md
// Activates for: ignition/**/*.py

Python 2.7 syntax only.
No f-strings, no walrus operator,
no type hints.

Use str.format() or % formatting.
Full module paths in imports.
Use system.kanoa.* APIs.

unichr(0x2705) not '✅'
// Jython Unicode limitation

Why this matters

Without path-scoped rules, Claude will write modern Python 3 for your Jython 2.7 codebase. You will correct it. It will apologize. Next session, it will do it again. Rules make the correction permanent and automatic.

Good candidates for rules

Language version constraints, API usage patterns, naming conventions, framework-specific idioms, and anything you have corrected more than twice. If you find yourself saying "no, we do it this way," that is a rule waiting to be written.

Key difference from CLAUDE.md: Rules are path-scoped and invisible. They load only when relevant files are being edited, so they cost nothing in sessions where you are not touching those files.

Layer 3

Skills: Reusable Capabilities

Skills are markdown files in .claude/skills/ that give Claude domain-specific knowledge and instructions on demand. They are like plugins: you activate them when you need a particular capability, and they stay dormant otherwise.

Reference skills

Load domain knowledge for a specific context. A Kanoa API skill loads function signatures, parameter formats, and common patterns so Claude writes correct API calls without searching documentation.

Examples: /kanoa-api, /kanoa-tags, /perspective-views, /ectobox-library, /ignition-theme

Process skills

Guide Claude through a multi-step workflow. A spec-developer skill interviews you about your idea, asks non-obvious questions, and produces a structured specification document at the end.

Examples: /spec-developer, /spec-developer-explore, /tech-comm, /mermaid-style

Navigation skills

Help Claude find things in a large repository. A project-index skill maps topics to file paths so Claude can locate the right reference document without trial and error.

Examples: /project-index

Output skills

Produce formatted deliverables. A brand skill ensures every HTML file follows the company design system, and a wiki skill formats content for Azure DevOps compatibility.

Examples: /ectobox-html-brand, /azure-devops-wiki, /mermaid-retrofit

Practical tip: Do not auto-invoke skills. Add a governance rule that Claude must state which skill it wants to use and wait for confirmation. This prevents wasted context from skills loading when you did not need them.

7
Layer 4

Memory: Cross-Session Persistence

Claude Code's auto-memory system writes small markdown files that persist between conversations. This is how Claude remembers your preferences, your project status, and the feedback you gave three weeks ago, without you having to repeat it.

Four memory types

TypeWhat it stores
UserYour role, expertise, preferences, domain background
FeedbackCorrections and confirmed approaches (both failures and successes)
ProjectStatus, decisions, deadlines, who is doing what
ReferencePointers to external systems (Jira boards, Grafana dashboards, Confluence spaces)

Auto-triggers: Memory entries can include trigger conditions. For example, "whenever a delivery milestone happens, remind the user to check for documentation rot." These fire automatically when the condition matches, without the user needing to remember.

What NOT to store in memory

Code patterns (read the code), git history (use git log), debugging solutions (the fix is in the commit), architecture details (read the files), or anything ephemeral to the current session.

The feedback loop

Memory is most powerful for feedback. When you correct Claude ("do not mock the database in these tests"), that correction becomes a memory entry with the reason attached. Future sessions apply the correction without you having to repeat it. Over time, Claude's behavior converges on your preferences instead of resetting to defaults every morning.

Memory index

A MEMORY.md file serves as the master index. It loads every session and points to individual memory files. Keep it under 200 lines, because everything in it is context overhead.

8
Layer 5

Workflows: Process as Code

Workflows are markdown documents that define how you develop software. They are not Claude-specific features, but they become powerful when Claude knows about them and follows them. The governance table in CLAUDE.md tells Claude which workflow to load for which type of task.

Process framework

A shared set of rules that every project follows: development phases (URS, FRS, Design, Build, Test, Deliver, Retrospective), sign-off gates with objective checklists, mandatory spec updates when requirements change mid-build, and a tiered lessons-learned system that promotes recurring insights into the framework itself.

Spec templates

Standardized document structures for different project types. An app spec template covers standalone tools. A requirements template covers customer-facing Ignition projects. Both reference the process framework for shared rules, so you only maintain one set of quality gates.

Why workflows beat prompting

You could paste process instructions into every conversation. But a workflow file is version-controlled, reviewed at retrospectives, and improved over time. When Claude loads the workflow, it gets the latest version of your process, not a copy you made three months ago and forgot to update.

Example: The process framework includes "rationalization detection" for build phases. It flags phrases like "should work now" or "I'm confident this fixes it" as red flags that require evidence. This is a lesson learned from real projects, codified into the framework, and automatically applied whenever the framework loads.

How the Layers Compose

Each layer solves one problem. Together, they create a system where Claude has institutional knowledge without you managing it manually.

What a Real Session Looks Like

Here is what happens when you open Claude Code in a configured repository and say "add a scheduling feature to the Balchem project."

  1. CLAUDE.md loads automatically Claude reads the governance table and sees: "For Ignition customer features, load workflows/requirements_gathering_template.md." It does not load everything. It loads one file.
  2. Memory provides context The auto-memory index tells Claude that Balchem is a deployed project, that the user prefers direct feedback without cheerleading, and that the last doc audit was May 2026.
  3. Claude suggests a skill "I'd use /spec-developer to interview you through the requirements. Should I invoke it?" You confirm, and the skill loads an interview framework that asks non-obvious questions about the scheduling feature.
  4. Rules activate during coding When Claude starts writing Python for the Ignition gateway, the Jython rule auto-loads and enforces Python 2.7 syntax. The Kanoa rule ensures API calls use the correct parameter format. No correction needed.
  5. Workflow gates apply The requirements template includes sign-off checklists. Claude checks: "Every FR has an ID and priority? At least one negative test case? Interface table filled in for cross-component boundaries?" If not, it flags the gap before moving to build.
  6. Memory updates for next time At the end of the session, Claude saves relevant feedback and project status to memory. The next session picks up where this one left off.

Honest Assessment

Features We Evaluated and Deferred

Not everything Claude Code offers is worth using from day one. These are capabilities we looked at, understood, and chose to skip for now.

13

Available but Not Yet Needed

Custom Subagents

.claude/agents/ lets you define specialized AI agents with restricted tools and focused instructions. A "spec-reviewer" agent could check specs against quality gates using only read tools, preventing accidental edits.

Why we deferred: Solo workflow. The sign-off checklists in our process framework serve the same purpose with less complexity. Worth revisiting when the team grows.

Hooks

.claude/hooks.json lets you run shell commands before or after Claude uses specific tools. You could auto-validate frontmatter tags on every file write, or run a linter after every code edit.

Why we deferred: Hooks add debugging surface area. Our memory-based auto-triggers achieve similar enforcement with less operational risk.

@import in CLAUDE.md

CLAUDE.md files can @import ./path/to/file.md to compose context from multiple files without duplication. Useful when the root file grows past 200 lines.

Why we deferred: Our root CLAUDE.md is 118 lines. No need to split what already fits.

CLAUDE.local.md

A gitignored CLAUDE.md for machine-specific configuration (local paths, Docker host settings, personal preferences) that should not be shared with the team.

Why we deferred: Solo workflow on one machine. Becomes important when multiple developers share the repository.

14
Takeaways

What We Would Do First if Starting Over

We built all five layers over three months of real project work. If we were starting fresh, this is the order we would follow based on where we saw the most impact.

  1. Start with CLAUDE.md Write a directory map and three to five hard rules. This alone eliminates the "Claude does not know where anything is" problem. Keep it under 200 lines.
  2. Add path-scoped rules for repeat corrections Every time you correct Claude's coding style more than twice, write a rule in .claude/rules/. Scope it to the file pattern where it applies. This eliminates an entire category of back-and-forth.
  3. Create skills for domain knowledge If you have API documentation, framework conventions, or process workflows that Claude needs for specific tasks, package them as skills. Start with reference skills (they are just markdown files with structured information), then graduate to process skills as your workflow matures.
  4. Let memory accumulate naturally Do not pre-populate memory. Let it grow from real sessions. When Claude gets something wrong and you correct it, that correction becomes a feedback memory. Over time, the system learns your preferences organically.
  5. Codify your process last Once you have patterns that work (a spec template that prevents rework, a testing checklist that catches real bugs), write them as workflow documents and reference them from CLAUDE.md. Process is the last layer because it needs to be grounded in experience, not theory.
